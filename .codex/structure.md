# Codex Structure and Workflow Guide

Last reviewed: 2026-06-17

Use this guide to understand how this VS Code theme extension is shaped, how files relate to each other, and where future changes belong. For official API links and version-specific references, read `.codex/docs.md` first.

## Mental Model

This repository is not an MVC app, a Vue app, or a component-based frontend. It is the Kawaii VS Code Color VS Code color theme with an optional settings-page Neon Effect patcher.

The project has four cooperating layers:

| Layer | Responsibility | Main inputs | Main outputs |
| --- | --- | --- | --- |
| Theme build layer | Preserves the base theme and applies Kawaii-specific overrides before VS Code loads the theme | Protected base theme, editable override theme | `themes/kawaii_synthwave-generated-color-theme.json` |
| VS Code manifest layer | Declares extension identity, generated theme contribution, commands, settings, activation rules, and entry point | `package.json` metadata and generated theme path | VS Code knows when and how to load the extension |
| Extension host layer | Handles command execution, reads settings, opens the setup webview, generates injected JS, patches VS Code workbench HTML, shows notifications | Command invocation, `kawaii_synthwave` settings, VS Code installation paths, user color settings | `neondreams.js`, patched workbench HTML, user messages, reload command, user color customization updates |
| Workbench renderer layer | Runs after VS Code reloads, waits for token CSS, transforms token styles, injects chrome CSS | Generated JS placeholders, active theme DOM, token CSS generated by VS Code | A new runtime `<style>` tag in the VS Code workbench DOM |

The normal theme still works through VS Code's public theme contribution. The glow effect is non-conventional because it modifies VS Code installation files and injects a script into VS Code's internal workbench.

## Primary Runtime Flow

Neon Effect enable path:

```text
Command Palette
  -> package.json activationEvents: onCommand:kawaii_synthwave.openSettings
  -> VS Code loads package.json main entry
  -> activate(context)
  -> registerCommand('kawaii_synthwave.openSettings')
  -> src/settings.js opens a WebviewPanel editor tab
  -> user opens Neon Effect from the side menu
  -> webview posts enable-neon to the extension host
  -> internal enableNeon() handler resolves VS Code workbench paths
  -> reads local CSS and JS template
  -> replaces template placeholders with settings-derived values
  -> writes neondreams.js into VS Code workbench folder
  -> patches workbench HTML with the marked script tag
  -> shows a VS Code notification
  -> runs workbench.action.reloadWindow
```

Renderer path after reload:

```text
VS Code loads patched workbench HTML
  -> neondreams.js runs in the workbench renderer
  -> MutationObserver waits until theme/token styles exist
  -> script confirms the active theme belongs to this extension
  -> script checks for expected token foreground colors
  -> token CSS is copied and optionally transformed into glow styles
  -> editor chrome CSS is appended
  -> generated <style id="kawaii_synthwave-theme-styles"> is added to document.body
```

Disable path:

```text
Kawaii VS Code Color: Settings webview
  -> user opens Neon Effect from the side menu
  -> webview posts disable-neon to the extension host
  -> internal disableNeon() handler
  -> resolves the same workbench HTML path
  -> removes the marked script tag from workbench HTML
  -> shows a notification
  -> reloads VS Code when needed
```

Disabling removes the HTML reference to `neondreams.js`. The current code does not delete the generated `neondreams.js` file.

Setup and color settings path:

```text
Command Palette
  -> package.json activationEvents: onCommand:kawaii_synthwave.openSettings
  -> activate(context)
  -> registerCommand('kawaii_synthwave.openSettings')
  -> src/settings.js opens a WebviewPanel editor tab
  -> Home page is shown first with theme and reference links
  -> side menu switches between Home, Settings, Color Settings, Neon Effect, Image Customization, Sync/Files, and Help
  -> Color Settings reads colors from themes/kawaii_synthwave-generated-color-theme.json
  -> user changes a color field
  -> webview posts update-color to the extension host
  -> extension host validates the hex color
  -> extension host writes a [Kawaii VS Code Color] block to user settings
  -> VS Code applies workbench.colorCustomizations or editor.tokenColorCustomizations live
```

## Inputs and Settings

Settings are declared in the manifest and read by the extension host:

| Setting | Declared in | Read by | Runtime effect |
| --- | --- | --- | --- |
| `kawaii_synthwave.brightness` | `contributes.configuration` | `workspace.getConfiguration("kawaii_synthwave")` | Clamped to `0..1`, converted to a hex alpha value, injected into glow CSS placeholders |
| `kawaii_synthwave.disableGlow` | `contributes.configuration` | `workspace.getConfiguration("kawaii_synthwave")` | Injected into the JS template; when true, token colors are not replaced with glow styles, but chrome CSS is still appended |

Settings flow:

```text
package.json contributes.configuration
  -> user settings.json
  -> activate(context) reads workspace configuration
  -> values are normalized in extension host
  -> placeholders in the JS template are replaced
  -> generated neondreams.js carries those values into the renderer
```

Important behavior:

- Settings are currently read once during extension activation, before command handlers run.
- If live setting updates are added, either read configuration inside the command handler or add a configuration-change listener.
- Brightness only controls glow alpha placeholders. It does not change the base theme palette.

## Communication Boundaries

There are two runtime environments:

| Environment | Can access | Cannot assume |
| --- | --- | --- |
| Extension host | VS Code extension API, Node `fs`, Node `path`, user settings, notifications, command execution | Direct DOM access to the editor workbench |
| Workbench renderer injection | Browser DOM, generated CSS strings, workbench HTML/CSS state | VS Code extension API, Node modules, direct calls back to the extension host |

Communication is one-way:

```text
extension host -> generated neondreams.js -> workbench renderer
```

The extension host passes data into the renderer by replacing template placeholders:

- `[DISABLE_GLOW]`
- `[CHROME_STYLES]`
- `[NEON_BRIGHTNESS]`

Do not add a new runtime dependency between injected JS and extension host unless there is a clear architecture change. If bidirectional communication is needed later, design it explicitly instead of relying on workbench globals.

User communication is handled through VS Code notifications:

- Success or refresh: `showInformationMessage`
- Expected permission/file-access failures: user-facing information message
- Missing workbench structure or generic failures: error message

Injected renderer communication is currently limited to `console.log`.

## Folder and Responsibility Rules

Keep changes inside the existing responsibility boundaries.

| Area | What belongs there | What should not go there |
| --- | --- | --- |
| Root manifest/package files | Extension identity, commands, settings, theme contribution, marketplace metadata, package lock | Runtime implementation details that should live under `src` |
| `src` | Extension host code and source assets used to generate the runtime injection | Marketplace screenshots, published theme JSON, Codex docs |
| `src/js` | Source templates for generated renderer scripts | Extension host command handlers or Node filesystem code |
| `src/css` | CSS that the generated renderer script injects into the workbench | VS Code theme color definitions and TextMate token colors |
| `themes` | Protected base theme, editable Kawaii overrides, and generated VS Code theme output | Runtime patching code |
| `scripts` | Small repository build utilities such as the color theme merge script | Extension runtime code loaded by VS Code |
| `test/unit` | Node unit tests for dependency-free logic and extension-host helpers | VS Code Extension Host integration tests |
| `test/dom` | `jsdom` tests and helpers for the settings webview DOM, safe webview events, timers, and message behavior | Runtime webview source files |
| `test/integration` | VS Code Extension Development Host tests using `@vscode/test-cli` | Unit-only helper tests |
| `test/e2e` | ExTester/WebDriver tests that open a disposable real VS Code window, drive the Command Palette, and interact with the real settings webview iframe | Unit logic, fake-DOM-only assertions, or safe-suite clicks that apply the real Neon patch |
| `.vscode` | Local VS Code development/debug configuration | Extension source or published assets |
| `.codex` | Agent-facing project guides and reference docs | Extension runtime files |
| Root image assets | README/Marketplace icon, screenshots, banners | Generated runtime assets |
| Root legacy CSS files | Legacy/manual custom CSS assets from the upstream workflow | New Neon Dreams source of truth |

Current source of truth:

- VS Code configuration get/inspect/update target handling lives in `src/settingsStore.js`.

- Settings bundle creation/application, Settings Sync state, and JSON import/export actions live in `src/settingsBundle.js`.
- Deterministic effect/image persistence helpers live in `src/settingsEffectsPersistence.js`.
- Setup webview HTML generation lives in `src/settingsWebview.js`.
- Workbench path detection and marked HTML patch helpers live in `src/workbenchPatch.js`.
- Real VS Code E2E helpers live in `test/e2e/helpers/extester-app.js`.
- Reusable settings webview DOM test setup lives in `test/dom/settings-webview-helper.js`.
- Protected upstream/base palette and token rules live in `themes/kawaii_synthwave-color-theme.json`.

- Command registration, setting normalization, filesystem writes, and workbench patching live in the extension host entry point or extracted host modules.

Settings webview visual rule:

- The webview must consume editor-provided VS Code webview tokens such as `--vscode-editor-background`, `--vscode-foreground`, `--vscode-button-background`, and `--vscode-panel-border`.
- Do not add a separate hardcoded product palette for page surfaces, text, controls, panels, or states.
- Help content should continue to centralize repository, issue tracker, README/homepage, and publisher/contact metadata derived from project data.

## Where New Code Should Go

Use small modules only when they remove real coupling. This project is currently small, so do not create a large framework structure by default.

Recommended extraction boundaries if the runtime grows:

| New responsibility | Suggested location/pattern | Notes |
| --- | --- | --- |
| Settings parsing and validation | `src/config/...` or a small helper near the command handler | Keep default values aligned with `package.json` |
| Workbench path detection | `src/workbench/...` | Keep path candidates version-aware and testable |
| Workbench HTML patching | `src/workbench/...` | Centralize script marker insertion/removal |
| Generated JS assembly | `src/neon/...` or a focused helper | Keep placeholders documented and validated |
| Notification messages | A constants/helper module | Preserve user-facing clarity and avoid duplicated strings |
| Token replacement rules | Renderer template or a focused renderer data file | Keep in sync with theme token foreground colors |
| Settings UI and persistence | `src/settings.js`, `src/settingsPersistence.js`, `src/settingsStore.js`, `src/settingsColorService.js`, `src/settingsBundle.js`, `src/settingsEffectsPersistence.js` | Keep Home and Color Settings in the same setup webview; keep user overrides in VS Code settings, not generated theme files; keep deterministic persistence logic unit-testable without a VS Code Extension Host |

Avoid these unless the project is deliberately refactored:

- `components`, `views`, `pages`, or frontend app folders.
- MVC-style `models/controllers/views`.
- Build output folders such as `dist` or `out` without adding a build pipeline and updating `package.json.main`.
- New dependencies without updating `package-lock.json`, `.codex/docs.md`, and compatibility notes.

If TypeScript is introduced later:

- Add an explicit build flow.
- Move source to TypeScript deliberately instead of mixing random `.ts` and `.js` files.
- Update `package.json.main` to point to compiled output.
- Add exact `devDependencies`, scripts, and validation commands.
- Add TSDoc blocks for exported or non-trivial functions.

## Theme and Glow Coupling

The glow system is coupled to exact theme token colors.

Flow:

```text
themes/kawaii_synthwave-color-theme.json
  -> scripts/build-color-theme.js
  -> themes/kawaii_synthwave-color-theme-overrides.json
  -> themes/kawaii_synthwave-generated-color-theme.json tokenColors foreground values
  -> VS Code generates .vscode-tokens-styles CSS
  -> injected script searches for expected hex colors
  -> matching `color: #hex;` declarations are replaced with glow CSS
```

If a token foreground color changes in the theme, update the renderer token replacement map when that color should still glow. Otherwise the glow script may decide the expected theme styles are not ready or may skip replacements.

The renderer glow template supports separate dark and light token replacement maps. The matcher is intentionally tolerant of VS Code-generated CSS variants:

- Dark glow-driving token colors include `#fe4450`, `#ff7edb`, `#fede5d`, `#72f1b8`, and `#36f9f6`.
- Light glow-driving token colors include `#0000ff`, `#008000`, `#098658`, `#a31515`, `#cd3131`, `#811f3f`, `#800000`, `#ff0000`, `#0451a5`, `#000080`, and `#000000`.
- Token matching must allow normal hex colors, uppercase/lowercase values, optional opaque alpha suffixes such as `#000000ff`, and spacing variants such as `color:#0000ff;`.
- The injected `#kawaii_synthwave-theme-styles` style tag is not single-use. It must be updated when `.vscode-tokens-styles` changes so switching between Kawaii VS Code Color and Kawaii VS Code Color Light can rebuild the glow styles.
- Keep the observer bounded by signature checks, not by permanent disconnects. Reprocessing should happen only when token CSS content changes.

Rules for changing theme colors:

- Do not edit `themes/kawaii_synthwave-color-theme.json` for Kawaii changes; it is the protected base.
- Put workbench color overrides in `themes/kawaii_synthwave-color-theme-overrides.json.colors`.
- Put syntax token additions or overrides in `themes/kawaii_synthwave-color-theme-overrides.json.tokenColors`; matching rules replace base rules by `name` or `scope`, and new rules append.
- Run `npm run build:theme` after editing overrides so `themes/kawaii_synthwave-generated-color-theme.json` is refreshed.
- Do not edit `themes/kawaii_synthwave-generated-color-theme.json` manually; it is build output.
- Prefer public VS Code theme keys in the overrides file for normal UI and syntax changes.
- Use `src/settings.js` for setup state and webview orchestration; use the extracted settings modules for user-local live color changes. They should write theme-scoped settings under `[Kawaii VS Code Color]`, not mutate repository theme JSON files at runtime.
- Use injected CSS only for effects VS Code themes cannot express, such as text shadows and internal chrome decoration.
- Validate TextMate scopes with `Developer: Inspect Editor Tokens and Scopes`.
- Check semantic highlighting interactions when a language server is active.
- Do not assume `themes/*.json` is strict JSON-friendly for every tool; the protected base theme contains JSONC-style trailing commas and the build script normalizes the generated output.

## Limitations

Treat these as negation prompts. Do not cross these boundaries unless the user explicitly asks for an unsupported workaround and the risk is documented.

- Do not try to implement text glow, `text-shadow`, gradients, animations, arbitrary CSS selectors, DOM changes, layout changes, or editor chrome shape changes through the VS Code color theme JSON. Official color themes cover workbench colors, syntax token colors/styles, and semantic token styling only.
- Do not treat transparency as a general CSS opacity system. VS Code theme colors support alpha only as part of supported hex color formats such as `#RGBA` and `#RRGGBBAA`; this does not allow blur, backdrop filters, opacity transitions, layered overlays, or arbitrary transparent DOM effects through the theme JSON.
- Do not make a color fully opaque by default when the VS Code Theme Color reference says the color should not be opaque. Some colors sit over annotations or other editor UI and can hide information if alpha is ignored.
- Do not use theme JSON for CSS animations, transitions, keyframes, hover animations, pulsing effects, blinking cursors, timed glow changes, or animated gradients. Those are CSS/runtime concerns, not color theme capabilities.
- Do not use theme JSON for JavaScript behavior: no `setTimeout`, no `setInterval`, no `requestAnimationFrame`, no event listeners, no DOM observers, no async calls, no file scans, and no initialization lifecycle.
- Do not add time-based loops to `src/js/theme_template.js` as a workaround for theme limitations. If runtime waiting is required, prefer an event-driven observer that disconnects once the target state is reached, as the current `MutationObserver` flow does.
- Do not add unbounded `setInterval`, recursive `setTimeout`, permanent `requestAnimationFrame`, broad DOM rescans, or repeated full-document style scans in the injected renderer. VS Code workbench performance and stability are more important than animated effects.
- Do not add workspace, filesystem, extension-directory, or installed-extension scanning unless the feature explicitly requires extension host code and the scan is bounded, version-aware, and error-handled. A color theme file cannot scan anything.
- Do not assume the injected JS has a supported initialization contract with VS Code. It runs only because the extension patches workbench HTML, then waits for DOM/theme state opportunistically.
- Do not invent workbench color keys. Use only keys from the VS Code Theme Color reference unless adding a real `colors` contribution point with a deliberate extension API design.
- Do not expect `tokenColors` to understand AST-level meaning, project context, symbol resolution, or cross-file information. TextMate tokenization is grammar/scope based; semantic meaning comes from semantic token providers, usually language services.
- Do not assume `tokenColors` always wins. Semantic highlighting is applied on top of TextMate highlighting when enabled and can appear after language server analysis.
- Do not modify `themes/kawaii_synthwave-color-theme.json` for Kawaii palette work. It is preserved as the base theme.
- Do not modify `themes/kawaii_synthwave-generated-color-theme.json` by hand. It is generated by `npm run build:theme`.
- Do not expect VS Code to load `themes/kawaii_synthwave-color-theme.json` and `themes/kawaii_synthwave-color-theme-overrides.json` as a native cascade. VS Code receives one contributed theme path, so the cascade is implemented by the repository build script.
- Do not try to register a new per-user generated theme file at runtime. VS Code theme contributions are static manifest entries; local live customization belongs in `workbench.colorCustomizations`, `editor.tokenColorCustomizations`, and `editor.semanticTokenColorCustomizations`.
- Do not re-add direct commands or menu contributions for `Color Settings`, `Enable Neon Effect`, or `Disable Neon Effect`. The public command opens `Kawaii VS Code Color: Settings`; Color Settings and Neon Effect are internal pages selected from that webview's side menu.
- Do not use any theme JSON file to register commands, settings, activation events, views, keybindings, menus, or extension behavior. Those belong in `package.json` contribution points and extension host code.
- Do not use a color theme file to change file icons, product icons, activity bar icons, codicons, or custom UI surfaces. Color themes are not icon themes or webviews.
- Do not use theme JSON to add runtime logic, filesystem access, reload behavior, notifications, or command handling. Runtime behavior belongs in `src/extension.js` or extracted extension host modules.
- Do not expect normal theme changes to update Neon Dreams behavior automatically. The glow map in `src/js/theme_template.js` is coupled to specific foreground colors and must be updated when glow-driving token colors change.
- Do not use workbench HTML patching as a normal theming mechanism. It is an unsupported workaround used only for Neon Dreams effects that the official theme API cannot express.
- Do not hide or remove user-facing warnings around Neon Dreams patching. Users must know that enabling glow modifies installed VS Code workbench files.

Source basis:

- VS Code Color Theme docs define the public theme surface as workbench colors, syntax colors/styles, and semantic colors.
- VS Code Theme Color reference defines supported workbench color identifiers and supported hex color formats, including alpha-channel transparency through `#RGBA` and `#RRGGBBAA`.
- VS Code Syntax Highlight Guide states TextMate theming rules map scope selectors to foreground, bold, italic, and underline style properties.
- VS Code Semantic Highlight Guide states semantic tokens are applied on top of TextMate highlighting and are computed by language extensions/language servers.

## Workbench Patching Mechanics

This extension intentionally patches VS Code internals. Treat this as the highest-risk part of the project.

Current mechanics:

- The extension derives the VS Code app directory from `vscode.env.appRoot`.
- It searches known workbench locations under `app/out/vs/code`.
- It supports `electron-browser` and `electron-sandbox` workbench folders.
- It supports `workbench.html` and `workbench.esm.html`.
- It writes `neondreams.js` next to the workbench HTML.
- It inserts a versioned script tag before the closing `</html>` marker.
- The inserted script is wrapped with marker comments so it can be removed later.
- The gated E2E command `KAWAII_E2E_ALLOW_NEON_PATCH=1 npm run test:e2e:neon` validates this only inside `.vscode-test/extest-111-neon`.
- The gated E2E uses five separate VS Code launches: capture baseline/apply dstgroup patch, validate dstgroup runtime after full restart and apply an alternate image, validate the alternate image after full restart and reapply dstgroup, validate dstgroup after full restart and disable, then validate restored runtime after another full restart.
- The gated E2E captures screenshots of the default watermark, dstgroup no-tab logo, alternate no-tab image, and reverted dstgroup logo; it also checks runtime CSS data URLs and `--vscode-*` token usage.

Patch marker contract:

```html
<!-- KAWAII SYNTHWAVE --><script src="neondreams.js?v=<timestamp>"></script><!-- NEON DREAMS -->
```

Rules:

- Preserve the marker comments unless updating both enable and disable logic together.
- Keep the script URL versioned so workbench reload can pick up regenerated `neondreams.js` content.
- Keep path detection centralized.
- Never assume one VS Code internal workbench path works for all versions.
- Handle `ENOENT`, `EACCES`, and `EPERM` as expected operational failures.
- Expect VS Code updates to overwrite patched workbench files; users must re-enable the glow after updates.
- Expect VS Code to show an unsupported/corruption warning after core file modification.

## View and UI Rules

There is no custom extension view.

User-visible surfaces are:

- VS Code Color Theme picker entry from `contributes.themes`.
- Command Palette entries from `contributes.commands`.
- VS Code settings from `contributes.configuration`.
- VS Code notifications from command handlers.
- Setup editor tab from `window.createWebviewPanel`.
- Visual workbench/editor changes from theme JSON and injected CSS.

Do not add another Webview, Tree View, sidebar, or custom editor unless the feature explicitly requires a real UI surface. If a future feature does need a view, document the new view contribution in `package.json`, create a dedicated source folder, and update this structure guide.

## Error Handling and Resilience

Current pattern:

- Known user-facing messages are centralized.
- Filesystem work is wrapped in `try/catch`.
- Permission and missing-file failures are shown to users.
- Workbench path detection returns `null` instead of throwing.
- Command disposables are pushed into `context.subscriptions`.

When adding behavior:

- Keep failures user-visible and actionable.
- Include enough internal context for debugging if adding logs.
- Do not silently swallow filesystem, parsing, or path-resolution failures.
- Keep the editor reload explicit after any successful workbench patch.
- Avoid unbounded DOM observers, polling loops, retries, or recursive recovery.

## Development Workflow

Recommended order for changes:

1. Confirm package and VS Code engine constraints in `package.json`.
2. Check `.codex/docs.md` for the official docs relevant to the area being changed.
3. Decide which layer owns the change: manifest, extension host, renderer template, injected CSS, or theme JSON.
4. Make the smallest scoped edit in that layer.
5. If a public contribution changes, update `package.json`.
6. If a setting changes, update both manifest defaults and host-side parsing.
7. If theme colors or token colors change, edit the overrides file and run `npm run build:theme`.
8. If token colors change, check the glow replacement map.
9. If workbench patching changes, validate enable and disable flows together.
10. Test with an Extension Development Host when possible.
11. Be careful running the neon enable command on a primary VS Code install because it patches installed VS Code files.

Useful checks for this repository:

- `npm pkg get name version publisher dependencies devDependencies engines`
- `npm run test:check`
- `npm run test:unit`
- `npm run test:dom`
- `npm run test:integration`
- `npm run test:e2e`
- `npm test`
- `npm run test:all`
- `KAWAII_E2E_ALLOW_NEON_PATCH=1 npm run test:e2e:neon` only when validating disposable Neon patch behavior.
- `npm run build:theme`
- Manual inspection of `package.json` contribution points.
- Manual theme validation in VS Code with `Developer: Inspect Editor Tokens and Scopes`.

There is no TypeScript config or lint script. Tests are split across unit, DOM, and VS Code integration layers, and the only build step is the theme merge script.

## Packaging and Release Boundaries

Before packaging or publishing:

- Check `.vscodeignore` so required runtime assets are included and private/local files are excluded.
- Run `npm run build:theme` so the generated theme is current.
- Ensure `package.json.publisher`, `name`, `repository`, and README links match the intended publisher/fork.
- Ensure `icon.png`, `banner.png`, and screenshots still match README/Marketplace usage.

- Update `.codex/docs.md` if dependencies, engine requirements, publishing tooling, or official references change.
