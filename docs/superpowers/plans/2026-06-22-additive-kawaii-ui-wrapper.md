# Additive Kawaii UI Wrapper Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Make the workbench UI effects additive by adding a `kawaii-vscode-colors-ui` class at the highest viable VS Code workbench document level and scoping injected CSS under that class.

**Architecture:** Keep the workbench HTML patch minimal: it inserts only `kawaii-vscode-colors-ui.js`. The extension host writes two runtime files next to the VS Code workbench HTML: a small JS bootstrap and `kawaii-vscode-colors-ui.min.css`, compiled from Sass and filled with user configuration placeholders at enable time. The JS detects whether a Kawaii theme wrapper is active, adds/removes the wrapper class on the highest viable root (`document.documentElement`, falling back to `document.body`), injects a versioned stylesheet link, and generates only scoped token glow overrides instead of copying and rewriting the entire VS Code token stylesheet.

**Tech Stack:** TypeScript extension host, browser JavaScript runtime template, Sass CLI, Node test runner, ExTester gated E2E.

---

## Current Problem

`src/js/theme_template.js` currently reads `.vscode-tokens-styles`, replaces `color: #...;` declarations inside copied CSS text, and writes the resulting CSS into `<style id="kawaii_synthwave-theme-styles">`.

That does not mutate VS Code's original style tag, but the injected CSS uses VS Code's original selectors directly. In cascade terms, it behaves like a later stylesheet override:

```css
.mtk5 {
  color: #fffafd;
  text-shadow: 0 0 10px #fc1f2c73;
}
```

Target behavior is additive and scoped:

```css
.kawaii-vscode-colors-ui .mtk5 {
  color: #fffafd;
  text-shadow: 0 0 10px #fc1f2c73;
}
```

The runtime must use the existing Kawaii theme selectors only as an activation check. The `kawaii-vscode-colors-ui` class itself must be added to the highest viable root, preferably `document.documentElement` (`<html>`), with `document.body` as a fallback. It must not be limited to the theme wrapper element.

## Recommended Runtime Shape

Use two generated runtime files:

- `kawaii-vscode-colors-ui.js`: bootstrap script, still inserted by the workbench HTML patch.
- `kawaii-vscode-colors-ui.min.css`: configured CSS generated from Sass, written next to the script.

Trade-offs:

- Separate CSS is more inspectable, cacheable, and closer to the requested Sass pipeline.
- The JS still needs to generate dynamic token glow rules because VS Code token classes such as `.mtk5` are generated at runtime and cannot be fully known from static Sass.
- The Sass root can initially be a migration orchestrator. It can include native Sass partials inside `.kawaii-vscode-colors-ui`, and CSS-only legacy sources can be bridged through a generated Sass partial until they are migrated.

Migration rule for CSS/Sass sources:

```scss
@use "sass:meta";

.kawaii-vscode-colors-ui {
  @include meta.load-css("generated/editor-chrome.generated");
}
```

If a source already exists as Sass, import it through `meta.load-css` inside `.kawaii-vscode-colors-ui`. If a source exists only as CSS, the build script should generate an intermediate Sass partial by reading that CSS and transforming selectors efficiently. Root selectors that need to apply to the root element itself must become `&...`, not descendant selectors.

- [ ] **Step 4: Implement the build script**

Add `scripts/build-ui-css.ts`:

```ts
import fs = require("node:fs");
import path = require("node:path");
import sass = require("sass");

export interface BuildKawaiiUiCssOptions {
  readonly generatedPartialFile?: string;
  readonly legacyCssFile?: string;
  readonly outputFile: string;
  readonly sourceFile: string;
}

export function generateScopedCssBridge(css: string): string {
  return String(css || "").replace(
    /(^|})\s*([^{}@][^{}]*)\{/g,
    (_match, previousClose: string, selectorList: string) => {
      const scopedSelectors = selectorList
        .split(",")
        .map((selector) => selector.trim())
        .filter(Boolean)
        .map((selector) => selector.startsWith("[class~=") ? `&${selector}` : selector)
        .join(",\n");

      return `${previousClose ? `${previousClose}\n` : ""}${scopedSelectors} {`;
    }
  );
}

export function buildKawaiiUiCss(options: BuildKawaiiUiCssOptions): void {
  if (options.legacyCssFile && options.generatedPartialFile) {
    const bridgeCss = generateScopedCssBridge(fs.readFileSync(options.legacyCssFile, "utf8"));
    fs.mkdirSync(path.dirname(options.generatedPartialFile), { recursive: true });
    fs.writeFileSync(options.generatedPartialFile, `${bridgeCss.trim()}\n`);
  }

  const result = sass.compile(options.sourceFile, {
    style: "compressed"
  });

  fs.mkdirSync(path.dirname(options.outputFile), { recursive: true });
  fs.writeFileSync(options.outputFile, `${result.css}\n`);
}

export function buildDefaultKawaiiUiCss(workspaceRoot: string = path.resolve(__dirname, "..")): void {
  buildKawaiiUiCss({
    generatedPartialFile: path.join(workspaceRoot, "src", "scss", "generated", "_editor-chrome.generated.scss"),
    legacyCssFile: path.join(workspaceRoot, "src", "css", "editor_chrome.css"),
    sourceFile: path.join(workspaceRoot, "src", "scss", "kawaii-vscode-colors-ui.scss"),
    outputFile: path.join(workspaceRoot, "src", "css", "kawaii-vscode-colors-ui.min.css")
  });
}

if (require.main === module) {
  buildDefaultKawaiiUiCss();
}
```
